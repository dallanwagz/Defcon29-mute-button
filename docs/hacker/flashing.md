## Normal Flash Procedure

1. **Hold BUTTON4** (bottom-right button)
2. **Plug the badge into USB** while continuing to hold BUTTON4
3. **Release BUTTON4 immediately** once the drive mounts
   - The badge appears as a USB mass storage drive (like a USB thumb drive)
   - The top-left LED (LED1) blinks red
4. **Copy `DC29.uf2` to the mounted drive**
   - On macOS: drag the file in Finder, or `cp DC29.uf2 /Volumes/DEFCON29/`
   - On Windows: drag the file to the drive in Explorer
5. The drive disappears and the badge reboots automatically
6. Confirm the badge is running:
   - The startup LED sequence runs (red → green → blue sweep)
   - A CDC serial port appears (`/dev/tty.usbmodem*` on macOS)

---

## Critical Timing

**Release BUTTON4 immediately after the drive mounts.**

If you hold BUTTON4 across the reboot (i.e., keep it held while the drive unmounts and the badge reboots into new firmware), the badge gets stuck in DFU mode indefinitely. It looks identical to a firmware crash: the drive is still mounted, LED1 blinks red, nothing works.

**Fix for stuck DFU mode:**
1. Unplug the badge
2. Release all buttons
3. Start over from step 1

---

## EEPROM Persistence

Flashing does **not** reset EEPROM. Your keymaps, LED colors, and challenge/game data survive.

Only these actions reset EEPROM:
- Changing `FIRMWARE_VERSION` in `main.h` (triggers version mismatch on boot → full erase)
- Calling `reset_eeprom()` explicitly in firmware
- Erasing the RWW flash section manually

## Restoring Factory Firmware

Pre-compiled `.uf2` files for the original DC29 firmware are in `Firmware/Compiled/`:

| File | Badge type |
|------|-----------|
| `DC29.uf2` | Default / unspecified |
| `DC29Human3.uf2` | Human badge |
| `DC29Creator2.uf2` | Creator badge |
| `DC29Artist2.uf2` | Artist badge |
| `DC29Vendor2.uf2` | Vendor badge |
| `DC29Speaker2.uf2` | Speaker badge |
| `DC29Goon2.uf2` | Goon badge |
| `DC29Press2.uf2` | Press badge |

Flash the appropriate `.uf2` for your badge type.

## Verifying the Flash

After flashing, confirm:

1. **Startup sequence:** LED1→LED2→LED3→LED4, each cycling red→green→blue
2. **Serial port:** `ls /dev/tty.usbmodem*` shows a device
3. **HID keyboard:** Open a text editor; press each button; confirm keystrokes appear
4. **Monitor events:**
   ```bash
   dc29 monitor --port /dev/tty.usbmodem14201
   ```
   Press each button and confirm `BUTTON N pressed` events appear.

## DFU Mode Details

The badge enters DFU (Device Firmware Update) mode when BUTTON4 is held during USB enumeration. In DFU mode:

- Appears as a USB mass storage device ("DEFCON29" or similar)
- LED1 blinks red
- No CDC serial port
- No HID keyboard function
- The bootloader in the first 8 KB of flash handles the UF2 copy

The bootloader is at `0x0000`–`0x1FFF`. The application starts at `0x2000`. The `samd21g16b_flash.ld` linker script enforces the `0x2000` origin.

The bootloader itself is in `Firmware/Compiled/dc29boot.hex`. **You almost certainly do not need to reflash the bootloader.** Flashing it requires a JTAG/SWD programmer (J-Link or ATMEL-ICE).

---

## Flashing via SWD (Advanced)

If the badge is completely bricked (bootloader corrupted), you need an SWD programmer:

1. Use a Segger J-Link or Microchip ATMEL-ICE
2. Connect to the SWD header on the badge (see schematic)
3. In Microchip Studio: `Tools → Device Programming`
4. Flash `dc29boot.hex` to address `0x0000`
5. Flash your application `.hex` to address `0x2000`

This is only needed if the bootloader itself is damaged, which shouldn't happen during normal UF2 flashing.
