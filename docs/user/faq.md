## Teams pairing

**No pairing dialog appears when I run dc29 flow**

Pairing only works during an active Teams meeting. The `canPair` permission is false outside of a call, so Teams won't show the dialog. Join a meeting first, then run `dc29 flow -v`.

Also check:
- Is Elgato Stream Deck running? It holds port 8124 and blocks dc29. Run `killall "Stream Deck"`.
- Is DC29: MuteIndicator still in the Teams Allowed or Blocked list? Remove it entirely in Teams → Settings → Privacy → Third-party app API.
- Delete `~/.dc29_teams_token` if it exists from a previous pairing attempt.

**`timed out during opening handshake`**

Another process (usually Stream Deck) is holding port 8124. Run `lsof -i :8124` to see what's connected, then kill it.

**Teams is connected but LED 4 never changes**

Run `dc29 diagnose` to verify the badge is connected and responding. Also confirm you're in an active Teams meeting, not just Teams being open.

## LED and button issues

**Pressing a button triggers TWO things (double action)**

The EEPROM has a keymap stored that fires a HID keystroke at the OS level, while the bridge also fires its own action. Fix:

```bash
dc29 clear-keys
```

This zeros all EEPROM keymaps. Use `dc29 diagnose` first to see what's stored.

**LEDs turn off during a Teams reconnect**

This was a bug (fixed in the latest commit). Update to the latest code and reinstall.

**LED 1 in Outlook looks orange**

This is the warm red `(220, 35, 0)` positional color. The green component adds warmth. If you want pure red, configure it in `~/.config/dc29/config.toml` under `[outlook.colors]`.

**LEDs dance/flicker when I press buttons**

The firmware's button-press animation is enabled. This is suppressed automatically when a bridge takes focus. If you see it while a bridge is active, run `dc29 flow -v` and watch for errors in the bridge initialization.

## Connection issues

**Badge not found / serial port missing**

- Check that the badge is powered (SW5 switch ON if present)
- Try a different USB cable or port
- On Windows: check Device Manager for an unknown device needing the CDC driver (`Firmware/Source/DC29/atmel_devices_cdc.inf`)

**`dc29 flow` immediately exits or shows "port not found"**

Pass the port explicitly:

```bash
dc29 flow --port /dev/tty.usbmodem14201
```

Or set it in config:

```bash
dc29 config init
# then edit ~/.config/dc29/config.toml and set [badge].port
```

**Multiple dc29 processes fighting over the serial port**

Check for orphaned processes:

```bash
pgrep -la python3 | grep dc29
```

Kill them, then restart.

## App context / focus detection

**Outlook/Slack focus not detected**

macOS Accessibility permission may be missing. The `osascript` call that detects the focused app requires it:

1. System Settings → Privacy & Security → Accessibility
2. Enable your terminal app
3. Restart the terminal

**Focus detection is slow or misses some switches**

The focus poll runs every 500 ms. This is intentional to avoid overwhelming macOS System Events with concurrent requests. It cannot be made much faster without impacting system performance.

## macOS-specific

**Outlook B1 opened Exposé / Mission Control instead of deleting email**

The EEPROM had a conflicting keymap. Run `dc29 clear-keys` to fix. The Outlook bridge uses `Cmd+Backspace` (move to trash) — if you see anything else, the firmware keymap is interfering.

**pynput Accessibility warning in logs**

```
WARNING pynput not installed — toggle hotkey disabled
```

Install the hotkey extra:

```bash
pip install "dc29-badge[hotkey]"
```

Or grant Accessibility permission to your terminal app (see above).

## Resetting to defaults

**Reset all EEPROM keymaps to no-ops:**

```bash
dc29 clear-keys
```

**Reset LED colors:** Use the serial console menu (any serial terminal at 9600 baud → follow the menu).

**Full EEPROM reset:** Change `FIRMWARE_VERSION` in `Firmware/Source/DC29/src/main.h` and reflash. This wipes all keymap and LED color data.
